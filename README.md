## Практика 19 — Реляционная СУБД PostgreSQL

Реализовано REST API для управления пользователями с подключением к PostgreSQL через ORM Sequelize.

**Стек:** Node.js, Express, Sequelize, PostgreSQL, dotenv

**Эндпоинты:**

| Метод | Адрес | Описание |
|-------|-------|----------|
| POST | `/api/users` | Создание пользователя |
| GET | `/api/users` | Список пользователей |
| GET | `/api/users/:id` | Получение пользователя |
| PATCH | `/api/users/:id` | Обновление пользователя |
| DELETE | `/api/users/:id` | Удаление пользователя |

**Сущность User:** `id`, `first_name`, `last_name`, `age`, `created_at`, `updated_at`

Дополнительно реализован простой фронтенд (`index.html`) для взаимодействия с API через браузер.

## Практика 20 — NoSQL СУБД MongoDB

Реализовано аналогичное REST API с подключением к MongoDB через ODM Mongoose.

**Стек:** Node.js, Express, Mongoose, MongoDB, dotenv

Структура эндпоинтов идентична практике 19. Ключевое отличие — отсутствие жёсткой схемы, автоматическое создание коллекций, идентификатор `_id` типа ObjectId.

## Практика 21 — Кэширование с Redis

Расширено приложение из предыдущих практик: добавлено кэширование GET-запросов через Redis.

**Стек:** Node.js, Express, Redis (`redis`), JWT-аутентификация, ролевая модель

**Реализовано:**
- `cacheMiddleware` — проверяет наличие данных в Redis перед обращением к хранилищу
- `saveToCache` — сохраняет результат в Redis с TTL
- Инвалидация кэша при изменении данных (POST, PUT, DELETE)
- TTL: пользователи — 60 сек, товары — 600 сек

## Практика 22 — Балансировка нагрузки

Настроена балансировка нагрузки через Nginx и HAProxy для приложения из практики 21.

**Стек:** Node.js, Express, Nginx, HAProxy, Docker

**Реализовано:**
- Три экземпляра сервера на портах `3000`, `3001`, `3002`
- Nginx: алгоритм Round Robin, сервер `3002` как резервный (`backup`)
- Отказоустойчивость: `max_fails=2`, `fail_timeout=30s`
- HAProxy: альтернативный балансировщик на порту `8080` с параметром `check`

## Практика 23 — Контейнеризация с Docker

Приложение с балансировкой нагрузки из практики 22 упаковано в Docker-контейнеры и развёрнуто через Docker Compose.

**Стек:** Node.js, Express, Nginx, Docker, Docker Compose

**Структура проекта:**
```
23/
├── backend/
│   ├── server.js
│   ├── package.json
│   └── Dockerfile
├── nginx.conf
└── docker-compose.yml
```

**Реализовано:**
- `Dockerfile` на базе `node:18-alpine` для backend-сервиса
- Три идентичных контейнера (`backend1`, `backend2`, `backend3`) из одного образа, различаются переменной `SERVER_ID`
- Nginx как балансировщик: `backend3` — резервный, `max_fails=2`, `fail_timeout=30s`
- Общая сеть `app-network` — контейнеры обращаются друг к другу по именам сервисов
- Запуск одной командой: `docker compose up --build`

**Проверка балансировки:**
```bash
curl http://localhost/
# {"server":"backend-1"}
# {"server":"backend-2"}
# {"server":"backend-1"}
```
